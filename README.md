# 宏观经济与大类资产配置研究设计

面向内部投研与 FOF 团队的本地静态研究终端，用来把“理论框架、宏观情景、历史复盘、当前判断、资产配置建议”放进同一个可浏览、可维护、可持续扩充的前端系统中。

这不是一篇研究报告，而是一套研究展示终端。  
核心目标是把研究过程组织成一条完整路径：

- 先判断宏观位置
- 再选择分析方法
- 最后映射到具体资产配置动作

## 1. 项目定位

这个项目目前主要解决 4 件事：

- 沉淀经济周期与大类资产配置的理论知识库
- 把不同理论框架映射到统一的宏观情景
- 结合历史案例，解释“为什么这样判断、为什么这样配置”
- 在首页给出当前宏观判断与资产配置建议，并通过“研究导航”把首页、情景页、资产页串成闭环

当前系统适合：

- 宏观策略研究
- FOF / 投顾 / 大类资产配置研究
- 内部投委会或策略会展示
- 作为知识库持续积累和迭代

## 2. 当前终端结构

前端主要分为 7 个模块：

1. 当前宏观与配置首页
2. 理论框架库
3. 宏观情景与资产表现
4. 资产映射库
5. 历史阶段复盘
6. 宏观经济仪表盘
7. 配置方法论

其中，首页不是单独的一页摘要，而是整个系统的“入口页”。  
当前已经打通的逻辑是：

- 首页显示当前宏观判断
- 首页显示“数据驱动周期判断”与“数据校验”结果
- 首页在“框架使用情况”里显示当前判断最相关的“推荐入门框架”
- 首页显示当前研究导航
- 首页并排显示“手工研究路径”和“数据驱动路径建议”
- 首页显示“手工路径 vs 数据路径”的差异解释
- 点击研究导航节点可以跳到对应情景或资产
- 点击首页配置建议卡可以直接跳到资产映射卡
- 情景页会同时显示“当前研究情景”和“数据驱动情景”及对应推荐资产
- 资产卡会显示它对应当前研究导航的第几步，以及当前数据环境下的校准结论

也就是说，这套系统已经不是“分散的模块堆叠”，而是一套可执行的研究路径终端。

## 4. 关键数据文件说明

### `data/frameworks.json`

理论框架库。  
目前已经包含或扩展过的主框架包括：

- 美林投资时钟
- 货币信用框架
- 货币-信用-实体三元框架
- 库存周期
- 普林格经济周期六段论
- 四维宏观环境框架
- 朱格拉 / 康波周期
- 金融周期
- 领先指标与转折点识别
- 宏观因子风险预算
- 风险平价 / 全天候
- Black-Litterman 与组合优化

每个框架尽量按统一模板维护：

- 起源背景
- 核心思想
- 分析方法
- 状态划分
- 传导机制
- 典型资产表现
- 关键指标
- 本土应用
- 历史表现
- 适用场景
- 局限性
- 常见误用
- 参考来源

前端“理论框架库”页面当前也已经不是平铺列表，而是进一步做了三层收口：

- 按 `识别框架 / 验证框架 / 配置方法论` 三组展示
- 顶部提供一条“推荐阅读顺序”
- 首页侧栏会根据当前判断，自动挑出最相关的“推荐入门框架”

也就是说，框架库现在不仅能查，也开始承担“学习路径”和“阅读导航”的作用。

### `data/macro_scenarios.json`

宏观情景层。  
把多套理论框架映射成统一情景，例如：

- 增长上行 / 通胀回落
- 增长下行 / 通胀下行
- 宽货币 / 紧信用
- 宽货币 / 宽信用
- 紧货币 / 紧信用

每个情景通常维护：

- 宏观特征
- 领先指标
- 常见配置思路
- 对应资产视角
- 对应历史案例

### `data/assets.json`

资产映射库。  
从资产本身出发，解释为什么这个资产在某些宏观位置更占优。

目前资产卡已经支持展示：

- 核心驱动
- 关键观察点
- 适用情景
- 配置用途
- 历史规律
- 本土应用
- 季度资产排序
- 报告常见风险提示
- 主要风险
- 常见误用
- 参考来源

### `data/historical_cases.json`

历史案例库。  
用于支撑“理论框架 -> 情景判断 -> 资产表现”的回溯。

当前案例重点在：

- 资产价格先行、总量后确认
- 货币宽松先行、信用与实体滞后确认
- 中外经济周期错位下的资产表现

### `data/current_view.json`

当前判断首页使用的数据来源。  
这是日常维护里最重要的文件。

主要包含：

- 当前宏观结论
- 国内主线 / 海外主线
- 当前研究导航
- 当前使用的核心框架
- 当前研究情景
- 当前配置建议
- 风险提示

如果后面要频繁更新“当前观点”，优先改这个文件。  
但需要注意：当前前端展示的 `current_view` 已不完全等于源文件原文，`scripts/build_macro_terminal_bundle.py` 会在打包时自动补充：

- `data_validation`
  - 数据驱动结论与当前手工路径是否一致
- `allocation_calibration`
  - 数据驱动下的资产优先级和校准说明
- `data_driven_path`
  - 基于最新数据生成的一条研究路径建议
- `path_comparison`
  - 手工路径和数据路径的差异解释

也就是说，`current_view.json` 仍是“人工主判断层”，但最终首页内容已经是“人工判断 + 数据校验 + 路径比较”的融合结果。  
指标看板本身已经不再从 `current_view.json` 读取，而是完全由 `data/macro_dashboard.json` 驱动。

### `data/config.json`

用于放：

- tabs 配置
- 配置方法论
- 资产映射研究导航说明
- 配置方法研究导航说明

也就是偏“页面框架”和“引导语”的配置层。

### `data/external_reports.json`

外部研究报告结构化结果。  
由 `scripts/build_external_reports_data.py` 生成，主要包含：

- 报告标题、日期、机构、原 PDF 路径
- 自动抽取的摘要与要点
- 关键图表页与图片路径
- 理论框架 / 宏观情景 / 资产 / 方法论 / 历史案例的自动匹配结果
- 新知识点识别结果
- 处理耗时与缓存命中情况

它是“外部报告 -> 系统内支持卡片”的中间层，而不是简单的 PDF 清单。

### `data/external_reports_review.json`

外部报告人工校正层。  
用于覆盖自动抽取里最容易出错的部分，例如：

- 指定更合适的关键图表页
- 增加 `review_status` 与 `review_notes`
- 对损坏或质量较差的 PDF 禁用自动预览

前端里的外部报告卡片会直接消费这层校正信息，并支持“全部报告 / 仅已校正”筛选。

### `data/macro_dashboard.json`

宏观指标看板的结构化结果。  
由 `scripts/build_macro_dashboard_data.py` 生成，供前端“当前指标看板”使用。

当前已经不仅是简单的“指标列表”，而是包含：

- 指标覆盖率与源文件导入状态
- 增长 / 通胀 / 流动性 / 信用 / 商品 / 风险资产等主题看板
- 派生指标（如期限利差、信用利差、中美利差、M1-M2 剪刀差）
- `data_driven_view`
  - 把最新数据翻译成增长周期、美林时钟、货币信用、库存周期与资产含义

前端首页、情景页、资产页和指标页都会消费这层数据。

### `外部宏观数据/manual_macro_series_overrides.json`

用于补齐 Excel 导出里没有的少量宏观时间序列。  
当前由 `scripts/fetch_online_macro_overrides.py` 自动写入，也可手工维护。

这层文件现在同时承担两件事：

- 存放公开网页抓到的少量补充序列
- 作为本地缓存层，避免每次更新都重复请求同一批公开网页

也就是说，新数据会被并入本地；后续更新时优先做“增量补充”，而不是每次从零抓取。

## 5. 当前研究导航与数据驱动路径逻辑

这是当前终端最重要的产品化设计之一。

系统现在把研究过程拆成一条连续路径，例如：

`当前宏观判断 -> 宽货币 / 信用修复早期 -> 优先看利率债与黄金 -> 再评估权益扩仓时点`

这条路径会同步映射到：

- 首页
- 宏观情景页
- 资产映射库
- 首页配置建议卡

所以用户不是只看到一个静态结论，而是能顺着这条路径一路点进去。

当前已经升级成“双路径结构”：

- 手工研究路径
  - 由 `current_view.json` 维护，代表投研人员当前的主观研究顺序
- 数据驱动路径
  - 由 `macro_dashboard.json` 自动推导，代表最新导入数据更支持怎样的研究顺序

系统会在首页直接显示两条路径，并额外给出：

- 两条路径当前是“基本一致”还是“存在偏差”
- 共同优先资产有哪些
- 手工路径更强调什么
- 数据路径更强调什么
- 下一步更适合验证什么

这使得它不只是“研究导航”，而是“研究导航 + 数据校验 + 偏差解释”的闭环。

## 6. 数据驱动内容已经融入哪些页面

当前最新导入的增长、通胀、信用、库存、流动性、市场数据，已经不仅停留在“仪表盘”页面，而是进一步融入了系统主内容：

1. 首页
   - Hero 区域显示“数据校验”状态
   - 首页主摘要会追加一条“数据校验层显示”
   - 新增“数据驱动周期判断”
   - “框架使用情况”里新增“推荐入门框架”
   - 配置建议新增“数据驱动资产校准”
   - 研究导航新增“数据驱动路径建议”
   - 研究导航区域新增“手工路径 vs 数据路径”的差异解释
2. 理论框架库
   - 按“识别框架 / 验证框架 / 配置方法论”分组显示
   - 顶部新增“推荐阅读顺序”
   - 每张框架卡和详情页显示 `Step` 标签，便于顺着读
3. 宏观情景页
   - 情景切换按钮会额外标出“数据驱动情景”
   - 情景详情中会显示“数据驱动优先资产”
4. 资产映射页
   - 资产卡会显示“数据校准”标签和优先级
   - 当前数据路径优先资产和当前数据焦点资产会被单独高亮
5. 指标页
   - 不只是展示原始指标值，还会把数据翻译成增长周期、美林时钟、货币信用和库存周期判断

因此当前系统已经不只是“手工观点页 + 指标页”，而是“数据 -> 周期判断 -> 情景映射 -> 资产校准 -> 路径建议 -> 理论阅读路径”的联动终端。

## 7. 如何运行

### 方法一：最简单的方式

直接双击：

- [一键打开宏观配置终端.command](./一键打开宏观配置终端.command)

现在这个脚本只负责：

- 检查本地 bundle 是否存在
- 启动本地网页服务
- 打开浏览器

如果本地 bundle 不存在，才会自动先触发一次更新。  
这样可以保证“打开页面”尽量快，而不会因为在线补数或数据构建步骤而卡很久。

### 方法二：只更新数据

双击：

- [一键更新宏观配置数据.command](./一键更新宏观配置数据.command)

这个脚本负责完整更新链路：

- 重建外部研究报告结构化数据
- 在线补齐少量 Excel 缺失指标
- 校验外部宏观 Excel 导入状态
- 重建宏观看板与前端 bundle

### 方法三：命令行方式

在项目目录执行：

```bash
# 1) 在线补齐少量 Excel 缺失指标（带本地缓存）
python3 scripts/fetch_online_macro_overrides.py

# 2) 校验外部宏观 Excel 导出状态
python3 scripts/check_macro_excel_exports.py

# 3) 重建前端 bundle
python3 scripts/build_macro_terminal_bundle.py
```

然后再打开 `index.html`，或通过本地静态服务打开。

### 方法四：一键部署到 GitHub

直接双击：

- [一键部署到GitHub.command](./一键部署到GitHub.command)

这个脚本会自动做 4 件事：

- 重建一次前端 bundle，避免页面和数据包不同步
- `git fetch origin main`，先确认 GitHub 上没有比本地更新的提交
- 自动 `git add -A`、生成一条时间戳提交
- 推送到 `https://github.com/fifamy/macro-allocation-terminal`

脚本是保守模式：

- 如果远端比本地更新，会直接停止，不会强推覆盖
- 如果本地没有任何新改动，会直接提示无需部署
- 如果你已经本地提交过但还没推送，它会直接把这些提交推上去

部署成功后，公开访问地址是：

- `https://fifamy.github.io/macro-allocation-terminal/`

如果你改的是原始 Excel、PDF 或外部报告抽取链路，建议先双击：

- [一键更新宏观配置数据.command](./一键更新宏观配置数据.command)

等数据重建完成后，再执行部署。

## 8. 推荐维护顺序

如果你后续继续扩这个项目，建议按下面顺序维护：

1. 先补 `frameworks.json`
   - 把理论框架讲清楚
2. 再补 `macro_scenarios.json`
   - 把框架映射成统一情景
3. 再补 `assets.json`
   - 把情景映射成资产语言
4. 再补 `historical_cases.json`
   - 用案例证明框架和情景
5. 最后更新 `current_view.json`
   - 输出当前判断和配置建议

这样终端会保持：

- 理论
- 情景
- 资产
- 历史
- 当前判断

这 5 层结构始终一致。

## 9. 外部研究资料怎么用

当前项目里已经有两个外部资料入口：

- `research-task/`
  - 用于做线上材料检索、摘要、归档、来源登记
- `外部研究报告/`
  - 本地下载的 PDF / PPT / 研究材料

建议后续工作流是：

1. 新报告先进入 `research-task/` 做摘要和分类
2. 原 PDF 放进 `外部研究报告/`，由 `scripts/build_external_reports_data.py` 抽取成 `data/external_reports.json`
3. 对抽取结果里不准确的关键图、摘要或预览，改 `data/external_reports_review.json`
4. 适合沉淀成长期知识的内容，拆进：
   - `frameworks.json`
   - `macro_scenarios.json`
   - `assets.json`
   - `historical_cases.json`
5. 适合形成当前观点的内容，再并入 `current_view.json`

不要把整篇报告原封不动塞进前端。  
这个终端的定位是“结构化知识 + 当前判断”，不是 PDF 仓库。

## 10. 外部报告融入系统的当前做法

当前“外部研究报告 -> 系统框架”的链路分为三层：

1. 自动抽取层
   - `scripts/build_external_reports_data.py` 负责扫描本地 PDF，抽取正文、摘要、关键图表页、匹配标签与新知识点。
2. 人工校正层
   - `data/external_reports_review.json` 负责修正自动抽取里最容易出错的地方，例如封面页、目录页、低质量图页或损坏 PDF。
3. 前端消费层
   - `scripts/build_macro_terminal_bundle.py` 将外部报告支持信息挂到框架、情景、资产、历史复盘和方法论卡片中。

当前历史复盘页已经做了收口：

- 不再在卡片正文中强行展示难以辨认的小图
- 只保留“来源报告”与单一 PDF 入口
- 历史案例若需要固定图页，优先在数据层手工指定

当前外部报告卡片支持：

- 显示 `已校正 / 待修复 / 未校正` 状态
- 顶栏切换“全部报告 / 仅已校正”

## 11. 宏观在线补数与本地缓存机制

当前 `scripts/fetch_online_macro_overrides.py` 已经不是“每次全量从零抓取”的脚本，而是“在线增量补数 + 本地缓存复用”的机制：

1. 把公开网页抓到的少量缺失指标写入：
   - `外部宏观数据/manual_macro_series_overrides.json`
2. 后续再次更新时：
   - 如果今天已经抓过，直接沿用本地缓存
   - 如果本地缓存已经覆盖当前发布窗口，也直接跳过请求
   - 如果某个公开源今天已经失败过，也不再重复超时重试
3. 最终再把这层缓存并入 `macro_dashboard.json` 与前端 bundle

这意味着：

- 新导入的数据会持久保存在本地
- 后续更新主要做“补新值”，不会每次全量重抓
- 打开页面时也不会因为在线抓取而卡住

## 13. 外部报告校正的推荐流程

当你发现外部报告支持卡里的摘要、图页或预览不对时，推荐按这个顺序处理：

1. 先看 `data/external_reports.json`
   - 确认自动抽取出的 `summary`、`top_pages`、`history_matches` 等字段到底哪里不准。
2. 再改 `data/external_reports_review.json`
   - 用 `review_status`
   - `review_notes`
   - `preferred_page_number`
   - `preferred_label`
   - `suppress_preview`
   这些字段做人工校正。
3. 如需固定图片页
   - 先把目标页渲染到 `research-task/generated/external-reports/charts/<report_id>/`
4. 最后重跑：

```bash
.venv/bin/python scripts/build_external_reports_data.py --force
python3 scripts/build_macro_terminal_bundle.py
```

不要优先改前端样式去“遮掩”错误数据。  
优先修 `external_reports_review.json`，这样框架页、情景页、资产页、历史复盘页会一起受益。

## 14. 常见问题

### 1. “一键打开”是不是还会重新下载数据

不会。  
当前 `一键打开宏观配置终端.command` 已改成“快速打开本地页面”为主，不再默认同步跑整套更新链路。

如果要更新最新数据，请先双击：

- [一键更新宏观配置数据.command](./一键更新宏观配置数据.command)

### 2. `CX_PMI_MFG: skipped` 或 `HTTP 403`

这通常不是本地程序冲突，而是外部网站拒绝访问或网络暂时不可达。  
当前脚本已经做了降级处理：

- 如果本地已有缓存，会显示 `skipped`
- 并继续沿用 `外部宏观数据/manual_macro_series_overrides.json` 里的旧值
- 不会阻断后续 bundle 构建

只有“在线失败且本地没有缓存”时，才会显示真正的 `failed`。

### 3. 某个在线补数指标总是 `cached`

这通常不是 bug，而是缓存策略生效了。  
脚本会在以下情况下直接沿用本地缓存：

- 今天已经抓取过
- 本地缓存已经覆盖当前发布窗口
- 该源今天已经失败过，不再重复超时重试

这样做是为了保证更新和打开终端足够快。

### 4. 某份 PDF 预览被禁用

如果 `external_reports_review.json` 里把某份报告标成了 `needs_source_fix` 并开启 `suppress_preview`，  
说明这份 PDF 当前结构损坏、文本抽取质量太差，或者自动图页完全不可信。  
这种情况下宁可不显示图，也不要挂错图。

### 5. 历史复盘页为什么不再显示缩略图

历史复盘页现在优先展示“文字复盘 + 来源 PDF”，而不是在卡片里塞一张很难看清的小图。  
如果某个历史案例确实需要固定图页，优先在数据层手工指定，而不是在卡片里恢复默认缩略图。
